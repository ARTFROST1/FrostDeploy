## 1. Обзор

### 1.1. Философия UI

FrostDeploy — инструмент для разработчиков. Интерфейс должен быть **функциональным, честным и быстрым**. Каждый экран решает одну задачу. Никаких декоративных элементов без практической ценности.

**Три принципа:**

| Принцип | Описание | Пример |
|---|---|---|
| **Инструмент, не витрина** | UI показывает состояние системы, а не продаёт продукт. Метрики, статусы, логи — всё на виду. | Dashboard показывает CPU/RAM/Disk, а не абстрактные «всё работает» |
| **Минимум кликов** | Любое действие — не более 3 кликов от стартовой точки. Деплой — 1 клик из обзора проекта. | Кнопка «Deploy latest» прямо на карточке проекта |
| **Прозрачность процессов** | Пользователь всегда видит, что происходит: статус деплоя, шаги pipeline, ошибки — без чёрных ящиков. | SSE-лог деплоя в реальном времени с таймстампами каждого шага |

### 1.2. Целевая аудитория

Основная персона — **инди-разработчик Артём** (PRD, Персона A): знает JS/TS, базовый Linux, управляет 2–5 проектами на одном VDS. Ему не нужны сложные виджеты — нужен быстрый доступ к статусам и деплою.

Вторичные: фрилансер Марина (5–10 сайтов), малая команда StudioAlpha (3–4 сервиса).

### 1.3. Дизайн-принципы

1. **Dark-first** — тёмная тема по умолчанию (DevOps-инструменты работают в терминале; светлый UI режет глаза ночью).
2. **Density over white-space** — плотная компоновка без лишних отступов. Информация важнее «воздуха».
3. **Terminal-native** — логи, SHA-хеши, команды — в моноширинном шрифте. ANSI-цвета в логах деплоя.
4. **Status-driven** — каждый элемент UI отражает текущее состояние; цветовые индикаторы однозначны.
5. **Keyboard-friendly** — навигация по клавиатуре; горячие клавиши для частых действий.

## 2. Дизайн-система

### 2.1. Цветовая палитра

Тёмная тема, построенная на нейтральных серых с акцентными цветами для статусов.

**Базовые цвета (dark theme):**

| Токен | HSL | Hex | Назначение |
|---|---|---|---|
| `--background` | 240 10% 4% | `#09090b` | Основной фон приложения |
| `--card` | 240 10% 6% | `#0f0f12` | Фон карточек и панелей |
| `--sidebar` | 240 10% 5% | `#0c0c0e` | Фон боковой панели |
| `--popover` | 240 10% 8% | `#141418` | Фон выпадающих меню и тултипов |
| `--border` | 240 6% 16% | `#27272a` | Границы элементов |
| `--border-hover` | 240 6% 22% | `#37373b` | Границы при наведении |
| `--foreground` | 0 0% 98% | `#fafafa` | Основной текст |
| `--muted-foreground` | 240 5% 64% | `#a1a1aa` | Второстепенный текст |
| `--accent` | 217 91% 60% | `#3b82f6` | Акцент (blue-500): ссылки, активные элементы |
| `--primary` | 0 0% 98% | `#fafafa` | Основные кнопки (белый текст на тёмном фоне) |
| `--ring` | 217 91% 60% | `#3b82f6` | Обводка фокуса |

**Статус-цвета:**

| Статус | Токен | Tailwind-класс | Hex | Применение |
|---|---|---|---|---|
| Успех / Active | `--status-success` | `text-emerald-500` | `#10b981` | deployed, active, health OK |
| Сборка / In progress | `--status-building` | `text-blue-500` | `#3b82f6` | deploying, building, queued |

| Неактивен / Idle | `--status-idle` | `text-zinc-500` | `#71717a` | created, stopped, no deploys |

### 2.2. Типографика

Два семейства шрифтов: sans-serif для интерфейса, monospace для кода и логов.

| Назначение | Шрифт | Fallback | Размер (rem) | Line-height |
|---|---|---|---|---|
| **Заголовок страницы** | Inter | system-ui, sans-serif | 1.5 (24px) | 1.33 |
| **Заголовок секции** | Inter | system-ui, sans-serif | 1.125 (18px) | 1.44 |
| **Основной текст** | Inter | system-ui, sans-serif | 0.875 (14px) | 1.43 |
| **Мелкий текст** | Inter | system-ui, sans-serif | 0.75 (12px) | 1.33 |
| **Код / SHA / команды** | JetBrains Mono | ui-monospace, monospace | 0.8125 (13px) | 1.54 |
| **Лог деплоя** | JetBrains Mono | ui-monospace, monospace | 0.75 (12px) | 1.5 |

**Правила:**
- SHA-хеши: первые 7 символов, моноширинный, `text-muted-foreground`
- Домены: обычный шрифт, `text-accent` (ссылка)
- Команды (`npm run build`): моноширинный, `bg-zinc-800 px-1.5 py-0.5 rounded`
- Таймстампы: мелкий текст, `text-muted-foreground`, формат «2 мин назад» / «31 мар, 14:30»

## 3. Навигация и информационная архитектура

### 3.1. Структура навигации

```
┌─────────────────────────────────────────────────────────────┐
│                        FrostDeploy                          │
├─────────────┬───────────────────────────────────────────────┤
│  SIDEBAR    │              MAIN CONTENT                     │
│             │                                               │
│  [Logo]     │   ┌──────────────────────────────────────┐   │
│  FrostDeploy│   │  Breadcrumbs (если вложенность > 1)  │   │
│             │   ├──────────────────────────────────────┤   │
│  ─────────  │   │                                      │   │
│  Dashboard  │   │  Page Header (заголовок + actions)   │   │
│  ─────────  │   │                                      │   │
│  ПРОЕКТЫ    │   ├──────────────────────────────────────┤   │
│  > Proj 1   │   │                                      │   │
│    Proj 2   │   │  Page Content                        │   │
│    Proj 3   │   │                                      │   │
│  ─────────  │   │                                      │   │
│  [+ Новый]  │   │                                      │   │
│             │   │                                      │   │
│  ─────────  │   │                                      │   │
│  Настройки  │   └──────────────────────────────────────┘   │
└─────────────┴───────────────────────────────────────────────┘
```

**Sidebar состоит из:**
1. **Логотип** — «FrostDeploy» + иконка снежинки, ссылка на `/`
2. **Навигация** — Dashboard (ссылка на `/`)
3. **Список проектов** — динамический, с иконками статуса; клик → `/projects/:id`
4. **Кнопка «+ Новый»** — создание проекта, ведёт на `/projects/new`
5. **Настройки** — внизу sidebar, ведёт на `/settings`

### 3.2. Карта маршрутов

```
/                                    ← Dashboard
├── /login                           ← Авторизация (вне layout с sidebar)
├── /setup                           ← Мастер установки (вне layout, первый запуск)
├── /projects
│   ├── /new                         ← Создание проекта (wizard)
│   └── /:id                         ← Обзор проекта (вкладка Overview)
│       ├── /deploys                  ← История деплоев
│       │   └── /:deployId           ← Консоль деплоя (SSE-лог)
│       ├── /env                      ← Переменные окружения
│       ├── /logs                     ← Логи сервиса (journalctl)
│       └── /settings                 ← Настройки проекта
└── /settings                        ← Настройки платформы
```

**Маршруты без sidebar:** `/login`, `/setup` — full-screen layout.
**Маршруты с sidebar:** все остальные.

### 3.3. Breadcrumbs

Breadcrumbs отображаются при вложенности > 1 уровень от корня.

| Страница | Breadcrumb |
|---|---|
| Dashboard `/` | Не показываются |
| Проект `/projects/:id` | `Dashboard / LaVillaPine` |
| Деплои `/projects/:id/deploys` | `Dashboard / LaVillaPine / Деплои` |
| Консоль деплоя | `Dashboard / LaVillaPine / Деплои / #a1b2c3d` |
| Настройки проекта | `Dashboard / LaVillaPine / Настройки` |
| Настройки платформы `/settings` | `Dashboard / Настройки` |

### 3.4. Табы внутри страницы проекта

Страница проекта (`/projects/:id`) использует систему вкладок. Каждая вкладка — отдельный маршрут.

```
┌─────────────────────────────────────────────────────────┐
│  LaVillaPine                           [Deploy latest ▶]│
├─────────────────────────────────────────────────────────┤
│  [Обзор]  [Деплои]  [Env]  [Логи]  [Настройки]        │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  Контент активной вкладки                               │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

| Вкладка | Маршрут | Описание |
|---|---|---|
| Обзор | `/projects/:id` | Статус, метрики, последние коммиты, домен |
| Деплои | `/projects/:id/deploys` | История деплоев (таблица) |
| Env | `/projects/:id/env` | Редактор переменных окружения |
| Логи | `/projects/:id/logs` | Journalctl-логи сервиса |
| Настройки | `/projects/:id/settings` | Build/start команды, домен, порт, удаление |

### 4.1. Страница авторизации (`/login`)

**Назначение:** Single-user аутентификация по паролю (FR-600).

**Layout:** Полноэкранный, без sidebar. Центрированная форма.

```
┌─────────────────────────────────────────────────────────┐
│                                                         │
│                                                         │
│                                                         │
│                    ❄ FrostDeploy                        │
│                                                         │

│     Пароль:     │ ••••••••••••     │                    │
│                 └──────────────────┘                    │
│                                                         │
│                 [       Войти      ]                    │
│                                                         │
│                 Забыли пароль?                           │
│                 Сбросьте через CLI:                      │
│                 frostdeploy reset-password               │
│                                                         │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

## 5. User Flows

### 5.1. Flow: Первоначальная настройка (Fresh Install)

```
┌──────────────┐    ┌────────────────┐    ┌──────────────────┐
│ npx install  │───>│ Открыть        │───>│ /setup — Step 1: │
│ на VDS       │    │ deploy.host:9000│    │ Создать пароль   │
└──────────────┘    └────────────────┘    └────────┬─────────┘
                                                   │
                                                   ▼
┌──────────────┐    ┌────────────────┐    ┌──────────────────┐
│ /            │<───│ /setup — Step 3│<───│ /setup — Step 2: │
│ Пустой       │    │ Указать домен  │    │ Ввести GitHub    │
│ Dashboard    │    │ платформы      │    │ PAT-токен        │
└──────┬───────┘    └────────────────┘    └──────────────────┘
       │
       ▼
  Пользователь готов
  к добавлению проектов
```

**Пояснение:**
1. Установка FrostDeploy из CLI (`npx frostdeploy init`)
2. Открытие URL в браузере → автоматическое перенаправление на `/setup`
3. Wizard: пароль → GitHub PAT → домен платформы
4. POST `/api/setup` → создание settings, Caddy-конфиг
5. Redirect на `/` — пустой Dashboard, можно добавлять проекты

```
│ Dashboard│────>│ /projects/new │────>│ Step 1: Ввести      │
│ [+ Новый]│     │ Wizard открылся│    │ GitHub repo URL     │
└──────────┘     └───────────────┘     └──────────┬──────────┘
                                                   │
                                     POST /api/detect
                                                   │
                                                   ▼
┌───────────────────┐     ┌──────────────┐     ┌─────────────────────┐
│ Step 4: Review    │<────│ Step 3: Env  │<────│ Step 2: Config      │
│ Проверка данных   │     │ Добавить     │     │ name, build, start, │
│                   │     │ переменные   │     │ domain, port        │
└────────┬──────────┘     └──────────────┘     │ (предзаполнено)     │
         │                                      └─────────────────────┘
   POST /api/projects
         │
         ▼
┌─────────────────────┐     ┌─────────────────────────┐
│ Проект создан       │────>│ Автоматический первый   │
│ Redirect →          │     │ деплой → консоль SSE    │
```

3. Подтверждение/корректировка конфигурации

6. Redirect на консоль деплоя для наблюдения за процессом

### 5.3. Flow: Ручной деплой (Manual Trigger)

```
┌─────────────────┐     ┌──────────────────────┐
│ /projects/:id   │     │ Выбор действия:      │
│ Обзор проекта   │────>│                      │
└─────────────────┘     │ A) Deploy latest ▶   │
                        │ B) Deploy конкретный │
                        │    коммит из списка  │
                        └──────────┬───────────┘
                                   │
                     POST /api/projects/:id/deploy
                     { sha?: "abc1234" }
                                   │
                                   ▼
                        ┌──────────────────────┐
                        │ Если активный деплой: │
                        │ HTTP 409 → Toast:    │
                        │ «Деплой уже идёт»    │
                        └──────────┬───────────┘
                                   │ Если OK:
                                   ▼
┌─────────────────┐     ┌──────────────────────┐
│ Деплой завершён │<────│ /projects/:id/       │
│ ✅ success      │     │ deploys/:deployId    │
│ или ❌ failed   │     │ SSE-лог в реальном   │
└────────┬────────┘     │ времени              │
         │              └──────────────────────┘
         ▼
┌─────────────────┐
│ Toast:          │
│ «Деплой успешен»│
│ или «Ошибка...» │
└─────────────────┘
```

**Пояснение:**
1. На странице обзора проекта: кнопка «Deploy latest» или кнопка «Deploy» у конкретного коммита
2. POST → проверка per-project mutex → если занят, HTTP 409, toast с ошибкой
3. Если свободен → redirect на консоль деплоя с SSE-стримом
4. По завершению → toast с результатом, обновление статуса в sidebar

---

### 5.4. Flow: Откат (Rollback)

```
┌─────────────────┐     ┌──────────────────────┐
│ /projects/:id   │────>│ /projects/:id/deploys│
│ Обзор или       │     │ История деплоев      │
│ обнаружили      │     │                      │
│ проблему        │     │ Найти последний      │
└─────────────────┘     │ успешный деплой      │
                        └──────────┬───────────┘
                                   │
                                   ▼
                        ┌──────────────────────┐
                        │ Клик по строке       │
                        │ успешного деплоя     │
                        │ → Кнопка «Откатить  │
                        │   до этой версии»    │
                        └──────────┬───────────┘
                                   │
                     POST /api/projects/:id/deploy
                     { sha: "prev_sha", triggered_by: "rollback" }
                                   │
                                   ▼
                        ┌──────────────────────┐
                        │ Консоль деплоя       │
                        │ (тот же pipeline:    │
                        │  fetch → build →     │
                        │  sync → restart)     │
                        └──────────┬───────────┘
                                   │
                                   ▼
                        ┌──────────────────────┐
                        │ Откат завершён       │
                        │ current_sha обновлён │
                        │ Badge: ↩️ rollback    │
                        └──────────────────────┘
```

**Пояснение:**
1. Проблема обнаружена → переход к истории деплоев
2. Поиск последнего успешного деплоя (зелёный badge ✅)
3. Кнопка «Откатить» → запуск полного pipeline для предыдущего SHA
4. В истории деплоев откат помечается иконкой ↩️ (`triggered_by: 'rollback'`)

---

### 5.5. Flow: Просмотр логов деплоя

```
┌─────────────────────────────┐
│ Точки входа:                │
│                             │
│ A) Dashboard → последние    │
│    деплои → клик по строке  │
│                             │
│ B) /projects/:id →          │
│    вкладка «Деплои» →      │
│    клик по строке           │
│                             │
│ C) Автоматический redirect  │
│    после запуска деплоя     │
└──────────────┬──────────────┘
               │
               ▼
┌──────────────────────────────────┐
│  /projects/:id/deploys/:deployId │
│                                  │
│  ┌─ Активный деплой? ─────────┐ │
│  │                             │ │
│  │  ДА → SSE подключение      │ │
│  │  GET /api/projects/:id/     │ │
│  │      deploy/stream          │ │
│  │  Real-time лог              │ │
│  │                             │ │
│  │  НЕТ → Загрузка из БД     │ │
│  │  GET /api/projects/:id/     │ │
│  │      deployments/:deployId  │ │
│  │  Полный лог (поле logs)     │ │
│  └─────────────────────────────┘ │
└──────────────────────────────────┘
```

**Пояснение:**
1. Три точки входа для просмотра логов деплоя
2. Система определяет: деплой активен (SSE) или завершён (из БД)
3. Для активного — SSE-подключение, real-time обновления
4. Для завершённого — загрузка полного лога из поля `deployments.logs`

## 8. Доступность (a11y)

### 8.1. Стандарт

Целевой уровень: **WCAG 2.1 AA**.

### 8.2. Требования

| Категория | Требование | Реализация |
|---|---|---|
| **Контрастность** | Минимум 4.5:1 для текста, 3:1 для крупного текста | Все цвета проверены на dark background (#09090b) |
| **Клавиатура** | Все интерактивные элементы доступны через Tab | shadcn/ui + Radix (built-in focus management) |
| **Focus ring** | Видимая обводка фокуса на всех интерактивных элементах | `ring-2 ring-blue-500 ring-offset-2 ring-offset-background` |
| **ARIA-labels** | Все иконки-кнопки имеют aria-label | `<Button aria-label="Запустить деплой">` |
| **Роли** | Sidebar → `nav role="navigation"`, лог → `role="log"` | Семантические HTML-теги + ARIA |
| **Live regions** | Toast-уведомления, статус деплоя | `aria-live="polite"` для статусов, `aria-live="assertive"` для ошибок |
| **Альтернативный текст** | Иконки фреймворков имеют alt | `<img alt="Astro SSR" />` или `aria-label` |
| **Skip link** | Ссылка «Перейти к содержимому» для пропуска sidebar | Скрытая ссылка, видимая при фокусе |

### 8.3. Горячие клавиши

## 9. Состояния и обработка ошибок

### 9.1. Состояния загрузки (Loading)

Каждая страница имеет Skeleton-состояние при первой загрузке данных.

| Страница | Skeleton |
|---|---|
| Dashboard | 4 серых прямоугольника MetricCard + 3 прямоугольника ProjectCard |
| Обзор проекта | 4 info-карточки skeleton + список из 5 строк-скелетонов |
| Таблица деплоев | 10 строк-скелетонов (height = строка таблицы) |
| Консоль деплоя | Пустой терминал с мигающим курсором |
| Env-переменные | 3 строки-скелетона key-value |

**Правило:** Skeleton повторяет общую форму содержимого: если будет таблица — skeleton-строки; если карточки — skeleton-прямоугольники.

### 9.2. Пустые состояния (Empty States)

| Контекст | Сообщение | CTA |
|---|---|---|
| Dashboard без проектов | «Нет проектов. Добавьте первый проект для начала работы.» | Кнопка «+ Новый проект» |
| Деплои проекта пусты | «Ещё нет деплоев. Запустите первый деплой.» | Кнопка «Deploy latest» |
| Env-переменные пусты | «Переменные окружения не заданы.» | Кнопка «+ Добавить» |
| Логи сервиса пусты | «Нет записей в журнале. Сервис не запущен или не генерирует лог.» | — |
| Поиск по логам (0 результатов) | «Ничего не найдено по запросу «{query}».» | Кнопка «Сбросить фильтр» |

### 9.3. Состояния ошибок (Error States)

| Уровень | Компонент | Пример | Отображение |
|---|---|---|---|
| **Страница** | `Alert` (destructive) | 500 от API, сервер недоступен | Красный Alert вверху страницы, кнопка «Повторить» |
| **Секция** | Inline error | Не удалось загрузить коммиты (GitHub API) | Серый блок с сообщением ошибки внутри секции |
| **Поле формы** | `Input` + error text | Невалидный URL, порт занят | Красная рамка input + текст ошибки под полем |
| **Действие** | `Toast` | Деплой вернул 409, ошибка сохранения | Toast в правом нижнем углу (destructive variant) |
| **Сеть** | Overlay | Нет подключения к серверу | Полупрозрачный overlay «Нет связи с сервером. Переподключение...» |

### 9.4. Обработка SSE-разрыва

При потере SSE-подключения во время деплоя:
1. Попытка переподключения (exponential backoff: 1с → 2с → 4с → 8с → 16с)
2. Показ banner: «Подключение потеряно. Переподключение...»
3. При восстановлении: запрос полного лога через REST, синхронизация с SSE

### 9.5. HTTP-ошибки

| Код | Контекст | Действие UI |
|---|---|---|
| 401 | Сессия истекла | Redirect на `/login`, toast «Сессия истекла» |
| 404 | Проект/деплой не найден | Страница 404 с «Вернуться на Dashboard» |
| 409 | Деплой уже активен | Toast «Деплой проекта уже выполняется» |
| 429 | Rate limit (логин) | Сообщение «Слишком много попыток. Повторите через N секунд» |
| 500 | Ошибка сервера | Alert «Внутренняя ошибка сервера», кнопка «Повторить» |

## 10. Анимации и переходы

### 10.1. Принципы

- **Минимум, по делу** — анимации только для обратной связи и ориентации; никаких декоративных transitions
- **Быстрые** — длительность 150–300ms
- **Уважение настроек** — `prefers-reduced-motion: reduce` отключает все анимации

### 10.2. Каталог анимаций

| Элемент | Анимация | Длительность | Easing | Trigger |
|---|---|---|---|---|
| **Page transition** | Fade-in контента (opacity 0→1) | 150ms | `ease-out` | Смена маршрута |
| **Sidebar hover** | Background highlight | 150ms | `ease-in-out` | Hover на пункте меню |
| **StatusBadge (deploying)** | Пульсация иконки (scale 1→1.2→1) | 1500ms loop | `ease-in-out` | `status === 'deploying'` |
| **Toast** | Slide-in справа + fade-in | 200ms (in), 150ms (out) | `ease-out` (in), `ease-in` (out) | Появление/исчезновение |
| **Dialog** | Fade-in + scale(0.95→1) | 200ms | `ease-out` | Открытие |
| **Dialog backdrop** | Fade-in (opacity 0→0.5) | 200ms | `ease-out` | Открытие |
| **Skeleton** | Мерцание (opacity 0.5→1→0.5) | 1500ms loop | `ease-in-out` | Загрузка данных |
| **Deploy progress bar** | Width transition | 300ms | `ease-in-out` | Смена шага деплоя |
| **Sidebar collapse** | Width 240px→56px | 200ms | `ease-in-out` | Клик по toggle (планшет) |
| **DeployLog scroll** | Smooth scroll к новой строке | `smooth` | — | Новая строка лога |
| **Mobile sidebar** | Slide-in (translateX -100%→0) | 250ms | `ease-out` | Открытие гамбургером |

### 10.3. Reduced motion

При `prefers-reduced-motion: reduce` все анимации заменяются:
- Transitions → мгновенная смена (0ms)
- Пульсация → статическая иконка
- Skeleton → статический серый блок (без мерцания)
- Slide-in → мгновенное появление

Реализация через Tailwind `motion-safe:` и `motion-reduce:` модификаторы.
