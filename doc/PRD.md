## 1. Обзор продукта

### Название

**FrostDeploy** — self-hosted универсальная платформа деплоя.

### Миссия

Предоставить инди-разработчикам и небольшим командам способ деплоить, управлять и мониторить любые веб-проекты через единый дашборд на собственном сервере — без зависимости от облачных PaaS и без необходимости Docker.

### Elevator Pitch

> **FrostDeploy** — мини-Vercel на вашем собственном VDS. Один независимый сервис заменяет все ваши самописные админки и ручной SSH-деплой. Автоматическое определение фреймворка, деплой по коммиту, мгновенный откат, real-time логи, встроенная аналитика — и всё это без Docker, на Node.js + SQLite + Caddy.

### Ключевые характеристики

| Характеристика | Значение |
|---|---|
| Тип продукта | Self-hosted open-source PaaS |
| Развёртывание | Независимый сервис на VDS (порт 9000) |
| Основной стек | Node.js + Hono/Fastify + SQLite + Caddy + systemd |
| Основная аудитория | Инди-разработчики, фрилансеры, малые команды |
| Лицензия | MIT (планируется) |

## 2. Проблема

### 2.1. Контекст

В двух существующих проектах — **LaVillaPine** (Astro SSR) и **SFOTKAI** (SolidStart) — реализованы собственные админ-панели для управления деплоем на VDS. Обе решают одну и ту же задачу: дать возможность деплоить код через веб-интерфейс без SSH.

### 2.2. Конкретные боли

| # | Проблема | Где проявляется | Влияние |
|---|---|---|---|
| 1 | **Дублирование логики деплоя** | LaVillaPine, SFOTKAI | Каждый новый проект требует написания админки с нуля. Одни и те же паттерны (git pull → build → rsync → restart) реализуются повторно. |
| 2 | **Проблема самоперезапуска** | LaVillaPine | Админка встроена в приложение. При деплое она перезапускает сама себя через `setTimeout(() => systemctl restart, 500ms)` — нестабильный workaround, а не архитектурное решение. |
| 3 | **Ручной SSH для нового проекта** | Любой новый проект | Добавление нового проекта на VDS требует ручных SSH-операций: настройка systemd-юнита, конфигурация Caddy, создание директорий, настройка DNS — каждый раз заново. |
| 4 | **Разрозненный мониторинг** | Все проекты | Нет единого места для отслеживания здоровья всех проектов: статуса процессов, использования ресурсов, аналитики трафика. |
| 5 | **Отсутствие стандартизации** | LaVillaPine vs SFOTKAI | Разные подходы к аутентификации (cookie vs JWT), мониторингу (proc vs API), логированию — усложняет поддержку. |

### 2.3. Существующие альтернативы и их ограничения

| Альтернатива | Ограничение для нашего случая |
|---|---|
| **Vercel / Render / Heroku** | Облачные платформы — не self-hosted, стоимость растёт с количеством проектов ($7–20+/мес за проект) |
| **Coolify** | Docker обязателен; тяжёлый стек (PostgreSQL + Redis + Soketi); минимум 2 CPU / 2 GB RAM только для платформы |
| **CapRover** | Docker Swarm обязателен; глобальный mutex (деплои строго последовательны); JSON для хранения данных (нет транзакций); устаревающий UI (jQuery) |
| **Ручной SSH** | Не масштабируется; нет UI; нет мониторинга; нет автоматизации |

### 2.4. Почему новое решение

Паттерны обоих существующих проектов (модель двух директорий, rsync, systemd, Caddy, git-based деплой) настолько совпадают, что извлечение их в отдельную платформу — естественный шаг эволюции. Ни одна из существующих open-source альтернатив не предлагает лёгкий self-hosted деплой без Docker.

| ID | User Story | Критерии приёмки |
|---|---|---|
| US-001 | Как **инди-разработчик**, я хочу **установить FrostDeploy на VDS одной командой**, чтобы **не тратить часы на настройку инфраструктуры**. | Установка через `npx frostdeploy init`: мастер первоначальной настройки запрашивает пароль, домен платформы, GitHub PAT. ### 4.2. Управление проектами

### 7.5. Схема базы данных MVP

```sql
-- Проекты
CREATE TABLE projects (
    id            TEXT PRIMARY KEY DEFAULT (lower(hex(randomblob(8)))),
    name          TEXT NOT NULL,
    repo_url      TEXT NOT NULL,
    branch        TEXT NOT NULL DEFAULT 'main',
    domain        TEXT UNIQUE,
    port          INTEGER UNIQUE NOT NULL,
    framework     TEXT,            -- 'astro-ssr', 'nextjs', 'static', etc.
    build_cmd     TEXT,            -- 'npm run build'
    start_cmd     TEXT,            -- 'node dist/server/entry.mjs'
    output_dir    TEXT,            -- 'dist/'
    env_vars      TEXT,            -- AES-256-GCM зашифрованный JSON
    src_dir       TEXT NOT NULL,   -- /var/www/{name}-src
    runtime_dir   TEXT NOT NULL,   -- /var/www/{name}
    service_name  TEXT NOT NULL,   -- systemd unit name
    current_sha   TEXT,
    status        TEXT NOT NULL DEFAULT 'created',
    created_at    TEXT NOT NULL DEFAULT (datetime('now')),
    updated_at    TEXT NOT NULL DEFAULT (datetime('now'))
);

-- История деплоев
CREATE TABLE deployments (
    id            TEXT PRIMARY KEY DEFAULT (lower(hex(randomblob(8)))),
    project_id    TEXT NOT NULL REFERENCES projects(id),
    commit_sha    TEXT NOT NULL,
    commit_msg    TEXT,
    status        TEXT NOT NULL DEFAULT 'pending',
    logs          TEXT,
    duration_ms   INTEGER,
    error         TEXT,
    triggered_by  TEXT DEFAULT 'manual',
    created_at    TEXT NOT NULL DEFAULT (datetime('now'))
);

-- Настройки платформы
CREATE TABLE settings (
    key           TEXT PRIMARY KEY,
    value         TEXT NOT NULL,   -- зашифрованные значения для секретов
    updated_at    TEXT NOT NULL DEFAULT (datetime('now'))
);

-- Индексы
CREATE INDEX idx_deployments_project ON deployments(project_id, created_at DESC);
CREATE INDEX idx_projects_domain ON projects(domain);
CREATE INDEX idx_projects_port ON projects(port);
```

| M-004 | Потребление ресурсов платформой (idle) | < 100 MB RAM, < 2% CPU | Чтение `/proc/meminfo` и `/proc/stat` для процесса FrostDeploy |

| M-006 | Время первоначальной установки | < 10 минут от ssh до работающего дашборда | Ручной замер |

## 11. Риски

### 11.1. Матрица рисков

| ID | Риск | Вероятность | Влияние | Приоритет | Митигация |
|---|---|:---:|:---:|:---:|---|
| R-001 | **Выполнение произвольного кода при сборке**: `npm install` выполняет postinstall-скрипты, которые могут быть вредоносными. | Средняя | Критическое | P0 | Сборка от непривилегированного пользователя; `npm ci --ignore-scripts`; в будущем — Docker-изоляция. |
| R-002 | **Исчерпание ресурсов одним проектом**: без контейнеров один процесс может занять всю RAM/CPU сервера. | Высокая | Высокое | P0 | systemd cgroups (`CPUQuota`, `MemoryMax`, `MemoryHigh`); алерты при превышении 80% CPU / 90% RAM. |
| R-003 | **Потеря данных SQLite**: при аварийном завершении или аппаратном сбое. | Низкая | Критическое | P0 | WAL-режим; автоматические бекапы через `.backup` (cron); конфигурация flush-интервала. |
| R-004 | **Поломка конфигурации Caddy**: невалидный конфиг может привести к отказу всего проксирования. | Средняя | Высокое | P1 | Validate-before-apply: всегда проверять конфиг перед применением; версионирование конфигов; откат к предыдущей версии. |
| R-005 | **Зависание сборки**: `npm ci` или `npm run build` может не завершиться (network issue, бесконечный цикл). | Средняя | Среднее | P1 | Таймаут на деплой (10 минут); автоматическое завершение процесса и снятие lock при таймауте. |
| R-006 | **Конфликт портов**: назначенный порт уже занят другим процессом. | Низкая | Среднее | P1 | Атомарное назначение через SQLite-транзакцию; проверка `lsof -i :{port}` перед назначением. |
| R-007 | **GitHub API rate limit**: 5000 запросов/час с PAT; при частом обновлении списка коммитов. | Низкая | Низкое | P2 | Кеширование ответов (60 сек); conditional requests (ETag/If-None-Match). |
| R-008 | **Dependency supply chain attack**: вредоносный пакет в npm-зависимостях проекта. | Средняя | Высокое | P1 | `npm ci --ignore-scripts`; в будущем — ограничение сети во время сборки (iptables/nftables). |

### 11.2. Продуктовые риски

## 12. Глоссарий

| Термин | Определение |
|---|---|
| **Build Engine** | Компонент FrostDeploy, отвечающий за полный pipeline сборки: от git clone до health check. |
| **Caddy** | Веб-сервер и reverse proxy с автоматическим HTTPS (ACME / Let's Encrypt), используемый FrostDeploy для маршрутизации трафика. |
| **Commit SHA** | Уникальный хеш-идентификатор git-коммита (обычно сокращённый до 7 символов), используемый для идентификации версии кода. |
| **Деплой** | Процесс доставки новой версии кода в рабочее состояние: от получения исходников до запуска обновлённого приложения. |
| **Детекция фреймворка** | Автоматическое определение типа проекта (Astro, Next.js, и т.д.) по содержимому package.json и конфигурационных файлов. |
| **Health check** | Проверка работоспособности приложения после перезапуска: HTTP GET на локальный порт, ожидается ответ 200. |
| **Модель двух директорий** | Архитектурный паттерн: src-директория (для git и сборки) и runtime-директория (для запуска), синхронизация через rsync. |
| **Mutex (per-project)** | Механизм блокировки, предотвращающий параллельные деплои одного проекта. Разные проекты могут деплоиться одновременно. |
| **Nixpacks** | Open-source инструмент (от Railway) для автоопределения языка проекта и генерации Docker-образа, используется FrostDeploy для поддержки не-Node.js языков. |
| **PAT (Personal Access Token)** | Токен GitHub для доступа к API (чтение репозиториев, коммитов, deployment status). |
| **Per-project lock** | Блокировка на уровне отдельного проекта (не глобальная) для предотвращения параллельных деплоев одного приложения. |
| **Proxy Manager** | Компонент FrostDeploy, управляющий конфигурацией Caddy: маппинг доменов на порты, SSL, валидация конфигов. |
| **rsync** | Утилита синхронизации файлов; используется для атомарной замены содержимого runtime-директории после успешной сборки. |
| **Setup Wizard** | Мастер первоначальной настройки FrostDeploy: запускается при первом запуске, запрашивает пароль, домен, PAT. |
| **SSE (Server-Sent Events)** | Протокол однонаправленного стриминга данных от сервера к клиенту по HTTP; используется для real-time логов деплоя. |
| **systemd** | Системный менеджер Linux для управления процессами; FrostDeploy создаёт systemd-юниты для каждого проекта. |
| **VDS (Virtual Dedicated Server)** | Виртуальный выделенный сервер, на котором запускается FrostDeploy и управляемые проекты. |
| **Validate-before-apply** | Паттерн проверки конфигурации перед применением (аналог `nginx -t`): если конфиг невалиден — откат, применение отменяется. |
| **WAL (Write-Ahead Logging)** | Режим работы SQLite, обеспечивающий concurrent reads и устойчивость к крашам. |
| **Zero-downtime deploy** | Деплой без простоя: новая версия запускается параллельно со старой, переключение происходит после health check. |
